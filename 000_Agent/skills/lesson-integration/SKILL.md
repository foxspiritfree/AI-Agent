# Lesson Integration (經驗萃取與規則整合)

本 skill 用於將 `000_Agent/memory/MEMORY.md` 中的「踩坑筆記」轉化為系統底層規則或 SOP，避免系統與未來的 Agent 重複犯錯。

## 執行流程 (Playbook)

### Step 1: 讀取輸入 (Ingest)
- 讀取 `000_Agent/memory/MEMORY.md` 的 `## 踩坑筆記` 區塊。
- 尋找尚未標記為 `[已整合]` 的新教訓。
- 如果沒有新教訓，回報使用者「目前無待整合的踩坑筆記」，並結束任務。

### Step 2: 分析與分流 (Triage)
對於每一則待處理的教訓，判斷其適用範圍：
1. **全域邊界**：這是否是跨工具、跨對話都必須遵守的安全、協作或路由原則？
   - 預期動作：準備修改 `000_Agent/CORE_RULES.md` 或 `000_Agent/DATA_ROUTING.md`。
2. **特定 SOP 優化**：這是否只是特定技術（如 GAS、Firebase、React）或任務的優化？
   - 預期動作：準備修改現有對應的 `000_Agent/skills/XXX/SKILL.md` 或外部框架的 SOP。
3. **新能力發展**：這個教訓是否揭示了一種全新的工作模式，且目前沒有技能涵蓋？
   - 預期動作：提議建立新的 skill。

### Step 3: 產出提案 (Propose)
- **不要直接修改任何系統檔案**。
- 產出一個「規則更新提案」，明確列出並 **詳細解說** 你的修改：
  - **來源教訓**：簡述踩坑內容。
  - **目標檔案**：預計修改或新增的檔案。
  - **具體文字與解說**：必須具體寫出「打算加入什麼文字」以及「這項教訓在系統中會長什麼樣子」。絕不能只用一句「將修改 XXX 檔案」帶過。
- **強制暫停**：列出提案後，詢問使用者：「是否同意以上修改？若同意，我將執行更新並在 MEMORY.md 標記這些教訓為已整合。」並等待指示。

### Step 4: 落地執行與清理 (Execute & Clean)
在獲得使用者批准後：
1. 執行所有提議的檔案修改。
2. 開啟 `000_Agent/memory/MEMORY.md`，將剛才處理完的教訓標題加上 `[已整合]` 標籤。
   - 例如：將 `### 冗餘 Workflows 的設計教訓` 改為 `### 冗餘 Workflows 的設計教訓 [已整合]`。
3. 簡短總結修改成果。
