# Problem Framing

把模糊情境整理成可行動的問題定義。這個 skill 的重點不是把所有可能性列完，而是找出目前最值得處理的那一個問題與下一步。

## 使用原則

- 先分清楚「現象」、「真正問題」、「目標」、「限制」。
- 把事實、推測、情緒判斷、他人說法分開。
- 優先找可驗證假設，不急著給大結論。
- 已經能判斷 trade-off 時，直接推薦一個下一步。
- 不把產品需求釐清取代 `pm`；如果情境明確是 PRD、feature scope、user story，改用 `pm`。

## 快速流程

1. 重述使用者想處理的事情，用一句話定義。
2. 拆成四格：
   - 現象：現在看得到什麼。
   - 目標：真正想改善或得到什麼。
   - 阻礙：卡住的限制、未知、衝突。
   - 假設：目前其實還沒被驗證的判斷。
3. 判斷問題類型：
   - 根因不明：用五個為什麼與現象/問題分離。
   - 選項太多：用目標、限制、代價篩掉選項。
   - 資訊不足：列最小驗證問題。
   - 行動卡住：縮小成下一個可做動作。
4. 輸出一個推薦與理由。

## 預設輸出

```markdown
**推薦**：[一個最值得先處理的問題或下一步]
理由：[一行理由]

**釐清**
- 現象：
- 目標：
- 阻礙：
- 假設：

**下一步**
1. [最小行動]
2. [需要驗證的問題]
```

## 問題類型處理

### 根因不明

- 先問「這是現象還是問題？」
- 對每個原因追問「所以呢？」確認它是否真的影響目標。
- 避免把「誰造成的」當作根因；優先找系統、流程、資訊、誘因或資源限制。

### 決策不明

- 先定義決策標準：目標、不可接受代價、時間限制、可逆性。
- 用期望值、最糟情境、可逆/不可逆來判斷。
- 若決策可逆，推薦小規模測試；若不可逆，先補關鍵資訊。

### 資訊不足

- 把未知分成：
  - 會改變決策的未知。
  - 不會改變決策的未知。
- 只追第一類未知。

### 行動卡住

- 把任務縮到可在短時間內完成的下一步。
- 若動機不可靠，改調整環境、提示、阻力與預先承諾。

## 與其他 skill 的邊界

- `pm`：產品需求、PRD、MVP、user story。
- `strategic-questioning`：需要設計問句、訪談、請教、對話。
- `ability-growth-planner`：目標是長期能力訓練。
- `notion-task-triage`：從 Notion 任務中排今日順序。
- `open-threads-to-notion`：把待釐清事項打包成 Notion 任務，不負責解題。
