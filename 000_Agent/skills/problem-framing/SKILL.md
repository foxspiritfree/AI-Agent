# Problem Framing

把模糊情境整理成可行動的問題定義。這個 skill 的重點不是把所有可能性列完，而是找出目前最值得處理的那一個問題與下一步。

## 使用原則

- 先分清楚「現象」、「真正問題」、「目標」、「限制」。
- 把事實、推測、情緒判斷、他人說法分開。
- 優先找可驗證假設，不急著給大結論。
- 真正值得處理的問題應該可控、可檢驗，且與目標有關。
- 已經能判斷 trade-off 時，直接推薦一個下一步。
- 不把產品需求釐清取代 `pm`；如果情境明確是 PRD、feature scope、user story，改用 `pm`。
- 若使用者提供優先度，先保留使用者優先度；技術依賴只能作為補充說明。
- 若問題涉及文案、語氣、等待、失敗回應、使用者情緒或體驗，必須把「使用者感受」列為問題定義的一部分。

## 快速流程

1. 重述使用者想處理的事情，用一句話定義。
2. 拆成四格：
   - 現象：現在看得到什麼。
   - 目標：真正想改善或得到什麼。
   - 阻礙：卡住的限制、未知、衝突。
   - 假設：目前其實還沒被驗證的判斷。
3. 若情境複雜，再拆成三層：
   - 現象：最先被看到或接收到的狀況。
   - 問題：哪個指標、結果或目標正在偏離。
   - 成因：哪些因素最可能造成指標偏離。
4. 判斷問題類型：
   - 根因不明：用五個為什麼與現象/問題分離。
   - 選項太多：用目標、限制、代價篩掉選項。
   - 資訊不足：列最小驗證問題。
   - 行動卡住：縮小成下一個可做動作。
5. 輸出一個推薦與理由。

## 預設輸出

```markdown
**推薦**：[一個最值得先處理的問題或下一步]
理由：[一行理由]

**釐清**
- 現象：
- 問題：
- 成因：
- 目標：
- 阻礙：
- 假設：

**下一步**
1. [最小行動]
2. [需要驗證的問題]
```

## 假說導向問題解決

在處理模糊或大型複雜問題時，不要直接進行漫無目的的資料蒐集，而應採用**假說導向 (Hypothesis-Driven)** 架構：

1. **建立初期假說 (Hypothesis Formulation)**：
   - 根據現有零星資訊與直覺，快速提出一個「最可能的問題根因」或「最可能的解決方案」作為假說。
   - 假說必須是「可被證偽 (Falsifiable)」的具體陳述，而非空泛的主觀臆測。
2. **MECE 議題拆解 (Issue Tree)**：
   - 將核心假說拆解為 2-4 個子議題，必須符合 **MECE 原則 (Mutually Exclusive, Collectively Exhaustive - 相互獨立、完全窮盡)**。
   - 確保子議題之間沒有重疊，且合起來剛好完整覆蓋了核心問題。
3. **BoE 快速驗證 (Back of the Envelope Calculation)**：
   - **數量級估算**：利用手邊僅有的核心數字或常識，在 5 分鐘內進行快速乘除（如信封背面計算），判斷假說是否在合理數量級內。
   - 如果估算結果與實際情況或預期偏差 10 倍以上，直接淘汰或修正該假說，避免投入大量時間做無效調查。
4. **有目的的資料蒐集 (Targeted Ingestion)**：
   - 只尋找「能證實或證偽該子假說」的關鍵證據。
   - 只要收集到足夠支持/反駁假說的關鍵數據，即停，立刻進行下一步行動，拒絕做資訊囤積狂。

## 問題類型處理

### 根因不明

- 先問「這是現象還是問題？」
- 若只有現象，先用 5W1H、正反例或時間軸補資訊：時間、行動、決策理由、誰受影響、哪裡變化。
- 將問題定義成指標、結果或目標偏離，而不是一句抱怨或表層需求。
- 對每個原因追問「所以呢？」確認它是否真的影響目標。
- 用五個為什麼、魚骨法或心智圖找候選成因，再用資料、案例或小測試驗證。
- 用 80/20 抓最可能影響結果的少數成因，避免平均處理所有原因。
- 避免把「誰造成的」當作根因；優先找系統、流程、資訊、誘因或資源限制。

### 決策不明

- 先定義決策標準：目標、不可接受代價、時間限制、可逆性。
- 用期望值、最糟情境、可逆/不可逆來判斷。
- 若決策可逆，推薦小規模測試；若不可逆，先補關鍵資訊。

### 資訊不足

- 把未知分成：
  - 會改變決策的未知。
  - 不會改變決策的未知。
- 只追第一類未知。

### 行動卡住

- 把任務縮到可在短時間內完成的下一步。
- 若動機不可靠，改調整環境、提示、阻力與預先承諾。

## 與其他 skill 的邊界

- `pm`：產品需求、PRD、MVP、user story。
- `strategic-questioning`：需要設計問句、訪談、請教、對話。
- `ability-growth-planner`：目標是長期能力訓練。
- `notion-task-triage`：從 Notion 任務中排今日順序。
- `open-threads-to-notion`：把待釐清事項打包成 Notion 任務，不負責解題。
