# 《新神鵰俠侶》對話閃過與連點跳過修復計畫 (v12)

針對《新神鵰俠侶》在現代高畫質 / dgVoodoo 環境下，劇情過場動畫（如陸家莊大廳黃蓉與郭靖對話）中出現的「點一下跳過多句對話」以及「連續對話瞬間閃過消失」問題，進行根源分析與全新 v12 修補方案。

## 1. 根源分析與歷史盲點

經過對 `ShenDiao.exe` 的深度逆向與除錯，我們發現了導致先前修改依舊失敗的致命成因：

1. **代碼洞（Code Cave）衝突與損毀**：
   - 之前的 v7 補丁（對話自動關閉延遲）與 v9 補丁（連續對話延遲）均使用了同一個記憶體位址 `0x52e861` 作為 Code Cave。這導致兩者的組合語言邏輯發生了嚴重覆蓋與損毀，在執行時會讀取到錯亂的暫存器與記憶體，造成功能完全失效。
2. **`Sleep` 造成的「滑鼠未釋放」死鎖**：
   - v9 嘗試在 `ShowDialogue` 發生覆寫時執行 `Sleep(1500)`。然而，`Sleep` 會凍結遊戲主執行緒（Main Thread），導致在此期間遊戲**無法處理滑鼠放開（WM_LBUTTONUP / DirectInput 釋放）事件**。當遊戲在 1.5 秒後醒來時，由於未接收到放開事件，系統依然認為滑鼠左鍵「處於按下狀態」，從而立即觸發下一句對話的跳過，造成「點一次吃掉好幾句對話」的嚴重惡性循環。
3. **滑鼠狀態判定遺漏 (State 3)**：
   - v11 雖然設計了邊緣觸發去抖器（Debouncer），但僅攔截了滑鼠剛按下的狀態（狀態 `1`）。當滑鼠按住稍微多出幾個毫秒時，遊戲輸入輪詢會將其判定為狀態 `3`（按住狀態）。由於 debouncer 未攔截狀態 `3`， dialogue manager 依然接收到了跳過事件，導致邊緣觸發機制失效。

---

## 2. 全新 v12 解決方案：顯示時間過濾器（Non-blocking Display Filter）

為了徹底、完美且不卡死遊戲地解決此問題，v12 拋棄了複雜的 DirectInput 邊緣檢測與會凍結執行緒的 `Sleep`，改為使用**非阻塞的顯示時間過濾器**：

### 核心機制
1. **Dialogue 啟動計時 (Cave 4)**：
   - 攔截 `ShowDialogue` (`0x4e21f8`)。當顯示全新對話時，調用 `GetTickCount()`，將當前系統時間存入全域變數 `[0x52e8F4]` 中，作為該句對話的「誕生時間」。
2. **滑鼠點擊時間過濾 (Cave 2)**：
   - 攔截滑鼠點擊判定 (`0x4f01d8`)。當滑鼠按下時，調用 `GetTickCount()` 並與 `[0x52e8F4]` 做差值計算：
     - 若**顯示未滿 500ms**，無條件忽略此點擊（回傳 `0`，不跳過）。
     - 若**顯示已滿 500ms**，才允許傳遞點擊（回傳 `1`，正常跳過）。
     - *這確保了玩家點擊第一句對話後，第二句對話在剛顯示的 500ms 內絕對不會接收到任何殘留的點擊事件，給予玩家充足的放開滑鼠時間！*
3. **連續對話防閃自動延遲 (Cave 3)**：
   - 攔截對話框覆寫邏輯 (`0x4e217f`)。當 `.ato` 腳本試圖在同一個 Slot 中快速覆寫尚未播完的 `wait_flag=0` 連續對話時：
     - 計算已顯示時間，若**未滿 800ms**，則補足時間（利用微小的 Sleep 讓舊對話在螢幕上停留至少 800ms，讓玩家有時間看清）。
     - 若**已滿 800ms**，則不延遲。
     - 更新 `[0x52e8F4]` 為新對話的誕生時間。

4. **乾淨重建 (Clean Rebuild)**：
   - 為了根治之前的 Cave 衝突與損毀，v12 將直接複製乾淨的原版 `ShenDiao.exe.vanilla.bak`，並重新**靜態一次性寫入**所有目前有效的補丁（免光碟、遇敵率極限 1%/0%、成長經驗 20%、30 FPS 限制、v12 對話過濾）。所有代碼洞均有精確的位址分配，絕無重疊衝突！

## 3. 使用者確認事項

> [!IMPORTANT]
> - 本補丁完全不需要修改 `.tkb` 對話資料庫（100% 避免了過場動畫卡死閃退的隱患）。
> - 本補丁為靜態二進位修補，不會引入 any 常駐外掛或記憶體注入工具。
> - 500ms 的保護時間既能完美防止連點跳過，又不會讓玩家感到操作延遲（0.5秒為人類正常閱讀反應時間）。
