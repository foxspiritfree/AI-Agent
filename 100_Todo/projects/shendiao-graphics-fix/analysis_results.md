# 《新神鵰俠侶》劇情對話跳過問題 - 逆向分析與失敗紀錄報告

本報告詳細記錄針對 2001 年昱泉版《新神鵰俠侶》在現代 Windows 系統與 dgVoodoo 環境下，「過場連續對話閃過」與「滑鼠單擊跳過多句」問題所進行的深度逆向工程分析、所套用的 12 代補丁演進、測試結果以及最終的技術結論。

---

## 1. 問題成因與現象

1. **滑鼠單擊連跳多句（連點吃字）**：
   - 玩家在 `wait_flag=1`（等待點擊）的對話中按下左鍵意圖進入下一句時，點擊事件會「穿透」並連續關閉隨後出現的數句對話，導致對話瞬間快進到最後一句。
2. **連續自動對話閃過（自動吃字）**：
   - 劇情過場動畫中（如陸家莊大廳郭靖與黃蓉的對話），角色說出連續的 `wait_flag=0`（自動播放）句子時，前幾句對話在螢幕上停留時間幾乎為零，瞬間被最後一句覆寫，玩家完全無法閱讀。
3. **劇情與探索對話之差異（關鍵觀測）**：
   - **此問題完全不會發生在劇情過場以外的對話**。也就是說，當玩家在**探索時期與地圖上的 NPC 進行自由對話時，完全不會產生任何跳對話或連點吃字的問題**。
   - 此現象進一步證實了第 3 節的技術結論：Bug 根源並非 Dialogue UI 渲染器或全域 Windows 滑鼠事件的通用問題，而是由 `.ato` 劇情腳本引擎在解析劇情腳本時，其獨立的 DirectInput 輪詢機制與高 CPU 影格率下的極速空轉所致。

## 2. 12 代修補演進與失敗分析紀錄
## 2. 15 代修補演進與失敗分析紀錄

為了解決此問題，我們前後設計並測試了 15 個版本的二進位修補方案，其技術原理與失敗原因如下：

| 版本 | 核心修補原理 | 測試結果 | 失敗原因分析 |
| :--- | :--- | :--- | :--- |
| **v1 - v4** | 修改 typing 浮點數常數（將 `0x130440` 從 `0.0011` 調至 `0.00022` 等），意圖改變逐字顯示的速度。 | **失敗** | 文字顯示速度被共享，極速打字雖然實現了瞬間顯示，但無法阻止腳本引擎在下一影格直接發送覆寫指令，對話依舊被瞬間切換。 |
| **v5** | 修改 tick check（將 `0xe3309` 與 `0xe34c5` 的延遲判定 `jb` 改成 `jmp`），防止遊戲在高格率/微幅卡頓下觸發 lag 自動跳過。 | **失敗** | 此 flag（`0x830d3c`）僅為遊戲「因卡頓被動跳過」的安全機制，並非玩家主動點擊或腳本引擎連續覆寫的根源。 |
| **v6** | 修改 `0x4f01ff` 的 `mov eax, 3` 為 `xor eax, eax`，停用滑鼠按住（Mouse Hold）的對話自動快進功能。 | **成功（局部）** | 成功移除了按住滑鼠時對話飛速前進的行為，但無法防止單擊左鍵產生的穿透快進。 |
| **v7** | 攔截 `HideDialogue` (`0x48c1cf`)，寫入 Cave 呼叫 `Sleep(1500)`，意圖在對話框關閉時強行延遲 1.5 秒。 | **失敗** | `.ato` 腳本引擎在連續說話時，並不會呼叫 `HideDialogue`，而是直接呼叫 `ShowDialogue` 在原 slot 進行覆寫，導致此延遲完全未被觸發。 |
| **v8** | 在對話更新主迴圈 `0xe32fd` 中寫入 `Sleep(33)`，強行將對話 UI 更新頻率鎖定在 30 FPS。 | **成功（局部）** | 穩定了對話更新的影格率，但無法解決點擊穿透與腳本高速覆寫的行為。 |
| **v9** | 攔截 `ShowDialogue` 覆寫點 (`0x4e217f`)，寫入 Cave 呼叫 `Sleep(1500)`，當 slot 被非空字串覆寫時強制延遲 1.5 秒。 | **失敗** | **滑鼠狀態鎖死**：`Sleep` 會凍結遊戲主執行緒，導致在此期間遊戲無法處理滑鼠放開的系統事件。當遊戲醒來時，仍判定滑鼠處於按下狀態，造成穿透惡化。同時此 Cave 與 v7 重疊衝突，導致程式碼損毀。 |
| **v10 - v11** | 攔截滑鼠點擊 `0x4f01a2`（`0x4f01c8` 與 `0x4f01d8`），在 `0x52e8f0` 變數中建立邊緣觸發去抖器（Edge-triggered Debouncer）。 | **失敗** | **滑鼠 Hold 判定遺漏**：輸入輪詢會將按住判定為狀態 `3`，此狀態繞過去抖器，依然觸發了跳過。同時因變數在唯讀 `.text` 段中寫入，高機率觸發 Windows 寫入保護閃退。 |
| **v12.1 - v12.3** | 在 PE 標頭中擴展 `.text` 虛擬大小，並將時間變數移至 `.data` 段可寫的 BSS 區域（`0x830f00`），實施非阻塞顯示時間過濾。 | **失敗（讀檔閃退）** | **變數空間碰撞**：`.data` 的 BSS 空間被遊戲大量的運行變數所共用。存檔載入或狀態重置時，`0x830f00` 的計時變數不斷被遊戲覆寫為 `0`，導致 elapsed time 計算失效。 |
| **v12.4** | **修改 PE 標頭屬性，將整個 `.text` 區段直接標記為可寫**。將計時變數放回我們專屬的 Cave 空間 `0x52e8f4`。 | **無閃退，但效果失敗** | **最終失敗根源**：即使計時變數完美運作、且 500ms 保護時間完全生效，**.ato 腳本引擎的指令執行依然繞過了 Dialogue UI Manager 的點擊過濾**。 |
| **v13** | 編譯 32 位元 `dinput.dll` 代理，在 DLL 載入時 Hook `CreateDevice`，並對 `GetDeviceState` 進行 COM VTable Hook (350ms 去抖)。 | **失敗** | **無滑鼠設備**：分析日誌發現 `ShenDiao.exe` 僅創立了一個控制器設備，根本沒有使用 DirectInput 獲取滑鼠點擊狀態。 |
| **v14** | 在 `dinput.dll` 中加入對 `ShenDiao.exe` 主進程 `user32.dll -> GetKeyState` 的 IAT Hook，攔截 `VK_LBUTTON`。 | **失敗** | **模組隔離**：IAT 劫持成功，但跳對話依舊。說明 `.ato` 腳本引擎判定代碼位於外部 DLL 中（如 `IFC21.dll`），或使用動態加載的 `GetAsyncKeyState`。 |
| **v15** | 升級 `dinput.dll` 全域 **Trampoline Inline Hook**（劫持 `GetKeyState`/`GetAsyncKeyState`/`PeekMessageA` 記憶體入口點），實施 150ms 去抖與 **QPC 高精度主迴圈 CPU 限流**。 | **失敗** | **底層時鐘超速**：QPC 限流成功將主迴圈空轉速度拉慢數百倍，去抖也極為靈敏，但劇情自動播放（`wait_flag=0`）依然閃過。這證明自動對話存在著與滑鼠無關、更為深層的腳本計時器硬編碼 Bug。 |

## 3. 最終逆向技術結論 (Architectural Summary)

經過 12 代補丁的極限測試與 PE 結構改造，我們終於定位了《新神鵰俠侶》對話跳過問題的**底層架構性根源**：

1. **腳本引擎與 UI 管理器的「雙重輪詢」機制**：
   - 遊戲使用雙執行緒或雙主迴圈架構。Dialogue UI Manager（`0x4e32b3`）負責渲染對話與處理 UI 點擊；而 `.ato` 腳本引擎（`0x49578d`）則獨立解析執行腳本指令。
   - 當對話為 `wait_flag=1` 時，`.ato` 腳本引擎**並不是單純等待 Dialogue UI 關閉**，它自身也在**直接調用 DirectInput 輪詢滑鼠狀態**。
   - 即使我們完美攔截並過濾了 Dialogue UI 的 `0x4f01a2` 點擊回傳，讓對話框 UI 不自我關閉，**.ato 腳本引擎自身依然偵測到了玩家按下滑鼠的實體事件**。腳本引擎直接判定「玩家已點擊」，從而強行執行下一條腳本指令，將新的對話內容硬塞進 `ShowDialogue`，產生了覆寫。這也是為什麼我們在 UI 層面做任何時間過濾或去抖，都無法阻止對話跳過的原因。
2. **高 CPU 影格率下的腳本極速空轉**：
   - 在 2001 年的 CPU（如 Pentium III 500MHz）上，腳本引擎更新頻率受到硬體制約，天然有數十毫秒的延遲，連點穿透與覆寫在慢速硬體下不易發生。
   - 在現代多核心數 GHz 的處理器上，腳本引擎執行一條指令僅需數微秒。一旦滑鼠事件未被硬體級攔截，腳本引擎會在一瞬間「空轉」執行完多條對話指令，將連續的對話光速覆寫。

## 4. 未來可能的研究方向 (For Future Reference)

若未來有其他逆向工程師 or Mod 製作團隊意圖再次挑戰此問題，建議放棄在「Dialogue UI 層面」的修改，轉向以下方向：
1. **硬體級滑鼠事件鉤子 (Global Mouse Hook)**：
   - 透過外掛 DLL（如 DInput 封裝器或 Win32 Hook）動態攔截並吞掉 `WM_LBUTTONDOWN` 或 DirectInput 輪詢，從硬體驅動層面為滑鼠按鍵製造物理延遲，防止其快速傳遞給 `ShenDiao.exe` 的腳本引擎。
2. **`.ato` 腳本編譯器改造**：
   - 深入逆向 `.ato` 的 Opcode 解析迴圈（`0x49585a` 附近），找出腳本引擎自身檢測滑鼠的指令分支，將腳本點擊檢測也與我們的時間計時器進行綁定。
